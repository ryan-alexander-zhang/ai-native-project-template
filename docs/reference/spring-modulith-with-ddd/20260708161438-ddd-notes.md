## Overview

Companion demo for an itnext.io blog series: a library-management modular
monolith on Spring Boot 4 / Spring Modulith 2 with DDD + jMolecules. Each
bounded context is a top-level package under `example`, communicating only
through domain events persisted via the Spring Modulith event publication
registry. Small and illustrative: two core business modules + cross-cutting
user-account/security, with fitness tests (`ApplicationModules.verify()` +
jMolecules ArchUnit rules) enforcing boundaries.

## Modules / bounded contexts

- `example.catalog` — cataloguing books into inventory (aggregate `CatalogBook`);
  **layered** flavor (`domain`/`application`/`ui`).
- `example.borrow` — holds, checkout, check-in (aggregates `Hold`, `Book`);
  **onion/hexagonal** flavor (`domain`/`application`/`infrastructure`).
- `example.useraccount` — cross-cutting module, `@ApplicationModule(displayName =
  "User Account")`, OAuth2/Keycloak JWT auth.
- Root `example.LibraryApplication` + `LibraryWebSecurityConfiguration` (OAuth2
  resource server).

## Spring Modulith boundary enforcement

- Package-per-module: one top-level package per context; Modulith infers modules
  from structure.
- Public API vs internals: cross-module types live in the module **base package**
  (e.g. event `example.catalog.BookAddedToCatalog` at the catalog root is the
  exposed API); `domain`/`application`/`infrastructure`/`ui` are internal
  sub-packages others can't reach into.
- `SpringModulithTests.verifyPackageConformity()` calls
  `ApplicationModules.of(LibraryApplication.class).verify()` — fails the build on
  illegal cross-package/cyclic deps.
- No explicit `allowedDependencies` whitelist; boundaries rest on default
  package-visibility rules + the single shared `@ApplicationModule`.

## Inter-module communication (events)

- Modules never call each other's services directly; borrow reacts to catalog via
  events. `CirculationDesk` listens for `BookAddedToCatalog` to create a local
  `Book` copy — **data replication across contexts**, not shared entities.
- Listeners use `@ApplicationModuleListener` (transactional, async,
  transaction-bound).
- Events raised from aggregates via Spring Data's
  `AbstractAggregateRoot.registerEvent(...)`, dispatched on save.
- Durable delivery via the Spring Modulith event publication registry backed by
  JPA (`spring-modulith-starter-jpa`) — at-least-once, transactional-outbox-style,
  in-process (no external broker).

## Tactical patterns & jMolecules

- Aggregates `CatalogBook` (`@AggregateRoot`, extends `AbstractAggregateRoot`),
  `Hold`, `Book` via static factories + command records (`AddBook`, `PlaceHold`,
  `Checkout`).
- VOs: `Barcode`, `Author`, `BookStatus` (jMolecules `ValueObject`), JPA
  `@Embedded`/`@EmbeddedId`; IDs as records implementing jMolecules `Identifier`
  (`BookId`, `HoldId`, `PatronId`).
- Domain events: records annotated `@org.jmolecules.event.annotation.DomainEvent`.
- Repos: `CatalogRepository`, `BookRepository`, `HoldRepository` (Spring Data
  JPA). App services `CatalogManagement`, `CirculationDesk` (`@Service
  @Transactional`).
- jMolecules layer annotations per module: catalog layered
  (`@DomainLayer`/`@ApplicationLayer`/`@InterfaceLayer`), borrow simplified onion
  (`@DomainRing`/`@ApplicationRing`/`@InfrastructureRing`). Deps: `jmolecules-ddd`,
  `-layered/-hexagonal/-onion-architecture`, `jmolecules-archunit`.

## Testing & docs generation

- `SpringModulithTests`: `verify()` for conformity + `new
  Documenter(modules).writeDocumentation()` to auto-generate C4 component diagrams
  (PlantUML) + module canvas docs.
- jMolecules ArchUnit fitness tests via `@AnalyzeClasses` running
  `JMoleculesDddRules.all()` and `JMoleculesArchitectureRules.ensureHexagonal()`.
- Module integration tests with `@ApplicationModuleTest` + the `Scenario` API:
  `scenario.stimulate(...).andWaitForEventOfType(...)`,
  `.publish(...).andWaitForStateChange(...)`. H2 runtime DB.
- OpenAPI/Swagger UI via springdoc.

## What to borrow (Java Spring Boot DDD template)

- Package-per-bounded-context (`app.<context>` top-level, `domain`/`application`/
  `infrastructure|ui` internals) and gate every build with
  `ApplicationModules.verify()`.
- Cross-context comms only via domain events + `@ApplicationModuleListener`,
  backed by the JPA event publication registry (transactional outbox, no broker);
  keep each context's own copy of data rather than sharing aggregates.
- Emit events from aggregate roots via `AbstractAggregateRoot.registerEvent(...)`
  on save; model commands/events/IDs/VOs as records.
- Layer intent with jMolecules annotations and enforce with jMolecules ArchUnit
  rules as executable architecture tests.
- Test modules in isolation with `@ApplicationModuleTest` + `Scenario`
  (`andWaitForEventOfType` / `andWaitForStateChange`) for deterministic async
  event choreography.
- Wire `Documenter.writeDocumentation()` into tests to keep C4/PlantUML diagrams
  generated from code.
