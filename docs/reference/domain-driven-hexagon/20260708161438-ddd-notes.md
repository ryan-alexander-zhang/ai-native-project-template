## Overview

A tutorial-style project whose README is a long-form article synthesizing DDD
tactical patterns, Hexagonal/Ports-and-Adapters, Onion, Clean Architecture, and
CQRS. Teaches an "Explicit Architecture" model (after Herbert Ograca): a
technology-agnostic Application Core surrounded by Adapters and Infrastructure,
code organized by vertical slices (one folder per use case).

## Layering & dependency rule

- Three concentric layers: **Application Core** (Domain + Application) · **Interface
  Adapters** (controllers, request/response DTOs) · **Infrastructure**
  (repositories, ORM models, external-service adapters).
- **Dependency Inversion**: dependencies point inwards; the Core stays
  framework/tech-agnostic.
- External resources reached only through **Ports** (interfaces in Core); Adapters
  in Infrastructure implement them, so technologies swap without touching domain.
- Enforce with tooling — Dependency Cruiser (TS) / **ArchUnit** (Java).

## Building blocks (responsibilities)

- **Entity**: identity + lifecycle; equality by id; behavior + invariants via
  methods (no public setters); valid on creation (fail fast).
- **Aggregate / Root**: cluster treated as one unit; root is the only entry point;
  only roots fetched from the DB; the transactional/consistency boundary.
- **Value Object**: immutable, no identity, structural equality; replaces
  primitives ("domain primitives" — Email, Address, Money) so illegal values can't
  exist.
- **Domain Event**: signals an in-process occurrence; decouples aggregates; handler
  changes committed in one transaction.
- **Domain Service**: process logic spanning multiple aggregates that fits no single
  Entity/VO; operates only on domain types.
- **Application Service (= Command Handler)**: orchestrates one use case, no domain
  logic itself; loads aggregates via Ports, invokes domain behavior, triggers
  side-effects via Ports; shouldn't call other application services.
- **Command / Query (CQRS)**: Command = state-changing intent, returns only id/
  metadata; Query = read intent, may bypass domain + repository. Optional
  command/query bus.
- **Port**: Core interface abstracting an external concern; "created to fit domain
  needs, not to mimic a tool's API".
- **Adapter**: Infrastructure implementation of a Port; also the Anti-Corruption
  Layer; bundles a mapper + external DTO + validator.
- **Repository**: data access for an aggregate; generic base + per-entity;
  implements a repository Port; not needed on read/query paths.
- **Mapper**: converts Domain Entity ↔ Persistence Model (and external ↔ domain).
- **DTO**: Request DTO = validated input contract; Response DTO = whitelisted
  output; kept separate from Commands for client backward-compat.
- **Persistence Model**: DB-optimized shape kept distinct from the Domain Entity
  (noted as possibly overkill for small apps).

## Command/query flow & use cases

- Lifecycle: Controller receives DTO → maps to Command/Query → Application Service/
  handler → loads aggregates via Repository Port, runs domain logic (optionally via
  Domain Services), calls Ports for side-effects → Infrastructure Adapter maps to
  persistence, does I/O, maps back → returns result → Controller maps to Response
  DTO.
- Reads can shortcut: query handlers may skip domain + repository and hit the DB
  directly (read model).
- **Vertical slicing**: one folder per use case (controller(s), request/response
  DTOs, command/query, handler); shared domain objects + repositories at module
  root (Common Closure Principle).
- Multiple entry adapters per use case are normal (http/cli/message/graphql), each
  mapping to the same Command/Query.

## Cross-cutting guidance

- **Validate vs Guard**: validate untrusted external input at the edge in Request
  DTOs (non-exceptional; return errors); guard invariants inside domain objects
  (exceptional; throw). Bad input isn't a bug; a broken invariant is.
- **Always-Valid Domain Model**: validate on construction and after each state
  change; immutable fields; no no-arg constructors; factories for complex creation;
  make illegal states unrepresentable.
- **Error handling**: model expected domain failures as typed errors via a
  Result/Either type (mapped to HTTP codes by the controller); throw only for
  non-recoverable technical faults.
- Aggregate invariants are the root's responsibility; cross-aggregate consistency
  via Domain Events within a transaction, not direct references.
- **Anti-Corruption Layer**: an Adapter + Mapper translating external/legacy models
  into the domain model.

## What to borrow (Java Spring Boot DDD template)

- **Package by feature/use case**, not by layer (`com.app.user.command.createuser`
  holding controller, DTOs, Command, handler); shared `domain`/`infrastructure` at
  module root (Common Closure Principle).
- **Framework-free Application Core**: Entities, Roots, VOs, Domain Events, Domain
  Services, and Port interfaces with zero Spring/JPA imports; confine
  `@RestController`, JPA `@Entity`, and adapters to Infrastructure/Interface.
- Enforce the dependency rule mechanically with **ArchUnit** (inner must not depend
  on outer) — the Java analogue of the repo's Dependency Cruiser setup.
- Split Domain Entity from JPA persistence model and translate with an explicit
  **Mapper** (MapStruct); expose repositories to the Core as domain **Port**
  interfaces with Spring Data implementations behind them.
- Adopt CQRS naming: `CreateUserCommand` + `CreateUserHandler` for writes (return
  only an id), thin read models/query handlers that may bypass repositories.
- **Two-tier validation**: Bean Validation (`@Valid`, `@Email`) on request DTOs at
  the edge; self-guarding domain objects that validate on construction/mutation and
  throw on invariant violation (no public setters; factories/builders).
- Model expected domain failures as **typed results** (a `Result`/`Either` or a
  sealed exception hierarchy) mapped to HTTP status in a `@ControllerAdvice`;
  replace primitives with Value Objects; propagate cross-aggregate changes via
  Spring domain events within a transaction.
