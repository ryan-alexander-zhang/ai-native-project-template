## Overview

A widely-used ASP.NET Core template operationalizing Clean Architecture
(≈ hexagonal / onion / ports-and-adapters). Ships a four-project layout + a
SharedKernel package and enforces the Dependency Rule by project references. It
is a **single bounded context** (one `ContributorAggregate`) with a largely CRUD
demo — stronger as a boundary/architecture exemplar than as rich tactical DDD.
Note: current `main` has moved off the "classic" choices — it now uses the
source-generated `Mediator` (not MediatR), **Vogen** for value objects, and
**FastEndpoints** (not MVC controllers).

## Project layout & dependency rule

- Four runtime projects under `src/`: `Core`, `UseCases`, `Infrastructure`,
  `Web`; plus `ServiceDefaults`/`AspireHost` and a `SharedKernel` package.
- Dependency direction (from `.csproj`): `Core` depends on nothing app-specific;
  `UseCases → Core`; `Infrastructure → Core + UseCases`; `Web → Infrastructure +
  UseCases`. Dependencies point inward; `Core` is the center.
- Ports live inward: domain-facing abstractions in `Core/Interfaces`
  (`IEmailSender`), repo/dispatcher abstractions in `Ardalis.SharedKernel`
  (`IRepository<T>`, `IReadRepository<T>`, `IDomainEventDispatcher`);
  implementations in `Infrastructure` (`EfRepository<T>`, `AppDbContext`).
  Read-model query-service ports declared in `UseCases`, implemented in
  `Infrastructure/Data/Queries`.
- A "Minimal" template (`min-clean`) collapses this to a single-project
  vertical-slice layout for small apps.

## Building blocks & libraries

- Aggregate/Entity base `EntityBase<TEntity,TId>` + marker `IAggregateRoot`
  (SharedKernel); `Contributor` has private setters + behavior methods.
- Value Objects via **Vogen** source generator (`[ValueObject<string>]`) with
  in-type `Validate`; enums via **Ardalis.SmartEnum**. (Library-generated, not
  hand-rolled DDD VOs.)
- Domain events raised via `RegisterDomainEvent(...)`; handlers co-located in
  `Core/.../Handlers` (domain behavior stays in Core).
- **Ardalis.Specification** for query criteria (`ContributorByIdSpec`);
  spec-aware repository.
- **Ardalis.GuardClauses** for argument/precondition checks.
- **Ardalis.Result** (`Result`, `Result<T>`, `Result.NotFound()`) as handler
  return type instead of throwing.
- Repository `IRepository<T>`/`IReadRepository<T>` implemented by `EfRepository<T>`
  constrained `where T : IAggregateRoot` (repo-per-aggregate).
- Use-case handlers `ICommandHandler<,>`/`IQueryHandler<,>` in
  `UseCases/Contributors/{Create,Update,...}`, dispatched via the **Mediator** bus.

## CQRS, events, validation, errors, persistence

- **CQRS**: commands/queries as messages + handlers; reads that don't need the
  aggregate bypass the repository via `...QueryService` read-model ports returning
  DTOs, keeping the write model clean.
- **Domain event dispatch — after commit**: an EF Core `SaveChangesInterceptor`
  collects tracked entities after `SavedChangesAsync` and calls
  `IDomainEventDispatcher.DispatchAndClearEvents`, so events fire only on
  successful persistence (in-process, **no outbox**).
- **Validation**: VO invariants inside Vogen `Validate`; preconditions via
  GuardClauses; docs favor pushing invariants into the domain.
- **Errors**: Result-over-exceptions for expected flow, mapped to HTTP by
  `Ardalis.Result.AspNetCore`.
- **Persistence**: EF Core behind Core-owned `IRepository` interfaces; configs,
  migrations, seed, and Vogen converters isolated in Infrastructure. Web uses
  FastEndpoints (REPR).

## Clean Architecture → DDD: what transfers (mapped to Java/Spring)

- `Core` → **domain** module: aggregates with behavior + invariants, domain
  events, specifications, domain services — kept framework-free (map, don't leak
  JPA/Spring in).
- `UseCases` handlers → **application** layer: command/query handlers as the
  explicit use-case boundary, returning an app-level result type.
- Ardalis.Specification → a domain **Specification** pattern (or Spring Data
  `Specification<T>` on the adapter side); express intent as named specs in the
  domain, translate in the adapter.
- Ardalis.Result → a `Result`/`Either` (Vavr `Either` or a small sealed
  `Result<T,Error>`) mapped to HTTP by a `@ControllerAdvice`.
- Ardalis.GuardClauses → a `Guard`/`checkRule` helper (DDD check-rule style) /
  Spring `Assert` for arguments.
- MediatR/Mediator → a command/query bus (thin dispatcher + handler interfaces),
  cross-cutting concerns as pipeline/decorator beans.
- EF repos behind interfaces → **ports & adapters**: domain-owned repository
  interfaces (one per root) implemented by Spring Data JPA adapters; keep the JPA
  entity separate from the domain aggregate.
- After-commit dispatch → Spring `@TransactionalEventListener(AFTER_COMMIT)` or
  Spring Data `AbstractAggregateRoot` — same "dispatch on successful save" idea.
- SharedKernel package → a `shared-kernel` module (base `AggregateRoot`, `Entity`,
  `DomainEvent`, `Result`, `Specification`, repository interfaces).

## Where it diverges from / underserves tactical DDD

- **Anemic/CRUD tilt**: `Contributor` is thin (name/status/phone), use cases are
  mostly CRUD — demonstrates boundaries far better than real aggregate invariants.
- **Single bounded context, NOT modular**: one aggregate namespace, one
  `AppDbContext`, one deployable; no module partitioning, context map, or
  per-module encapsulation. A layered clean-architecture template, not a modular
  monolith.
- **No integration-events / outbox story**: domain events are in-process only;
  no transactional outbox, no published-language/inter-module async contract —
  exactly the piece a modular monolith most needs.
- **Domain vs persistence model blurred**: the EF entity *is* the aggregate
  (persistence-aware domain); VOs are library-generated. Pragmatic, but not the
  pure-domain-vs-JPA-entity separation a Spring DDD template usually wants.
- **Strategic DDD absent**: no ubiquitous-language/subdomain/context-mapping
  guidance; "aggregate" is mostly a repository-scoping marker (`IAggregateRoot`).
- **Read side is a pragmatic bypass**: DTO query services are good CQRS ergonomics
  but not a DDD modeling pattern — don't mistake them for domain modeling.

## What to borrow (Java Spring Boot DDD template)

- Enforce the **Dependency Rule structurally**: modules `domain` (pure) /
  `application` / `infrastructure` / `web` + `shared-kernel`; forbid inward
  references with Gradle/Maven module deps + ArchUnit — as this template does with
  project references.
- Keep an **explicit application/use-case layer** with command & query handlers +
  a thin bus — Clean Architecture's biggest add over "just tactical DDD"; a
  natural seam for validation/logging/transaction decorators.
- **Result-as-value** for expected outcomes + one result→HTTP mapper; exceptions
  only for the exceptional.
- **Repository-per-aggregate ports** owned by the domain, Spring Data adapters
  behind them; a domain Specification pattern for query intent; read-side DTO
  projections separate from the write model.
- After-commit domain-event dispatch via `@TransactionalEventListener(AFTER_COMMIT)`
  — **then go beyond this template** by adding a transactional outbox + integration
  events and real module boundaries (Spring Modulith), which it omits.
- Treat CleanArchitecture as the **boundary skeleton** and add tactical-DDD
  richness yourself: rich aggregates, private setters, factories, guard/checkRule,
  real invariants — not the thin demo model.
