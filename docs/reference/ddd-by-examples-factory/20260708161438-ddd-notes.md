## Overview

A deliberately complete DDD reference for an enterprise manufacturing-factory
domain on the Spring stack (Boot 2.0, Data JPA, Data REST). A Gradle multi-module
modular monolith where the high-value logic (demand forecasting, shortage
prediction) is a rich hexagonal domain model, while CRUD-ish concerns go through
Spring Data REST. Derived from Event Storming (Big Picture → Design Level);
emphasizes "implement the domain model first, add infrastructure later".

## Bounded contexts & domain

- Problem: a factory must forecast demand, plan part deliveries, forecast stock,
  and predict/monitor material shortages so production isn't disrupted.
- Contexts as module pairs (`*-model` = domain, `*-adapters` = infra): Demand
  Forecasting (`io.dddbyexamples.factory.demand.forecasting`), Shortages
  Prediction (`...shortages.prediction`), Production Planning
  (`...production.planning`), Product Management (`...product.management`).
- `shared-kernel-model` holds cross-context types/events: `Demand`, `DailyId`,
  `RefNoId`, integration events `DemandedLevelsChanged`, `ReviewRequired`.
- Sub-domains as sub-packages, e.g. shortages `calculation`/`monitoring`/
  `notification`, demand `command`/`projection`/`persistence`.

## Tactical patterns in code

- **Aggregate root** — `ProductDemand` (package-private): coordinates
  `adjust/process/review`, guards internals, emits events only on real change.
- **Entity** — `DailyDemand` (identity `DailyId`, mutable state, returns a
  `Result` describing what changed).
- **Value Objects** — `Adjustment`, `Demand` (Lombok `@Value`, factory methods
  `strong()/weak()`); `DailyId`, `RefNoId` as identifier VOs in the shared kernel.
- **Domain Events** — `DemandedLevelsChanged`, `ReviewRequired`; shortages emits
  `NewShortage`/`ShortageSolved` via `ShortageEvents`.
- **Policy / invariant** — `ReviewPolicy` (`reviewNeeded(...)`) injected into
  `DailyDemand.update(...)`; rules live in the model, not adapters.
- **Application Service (primary port)** — `DemandService` is thin: load
  aggregate → call one domain method → save; shortages: `ShortagePredictionService`.
- **Repository port (secondary)** — `ProductDemandRepository` (package-private
  `get/save/initNewProduct`); adapters `ProductDemandORMRepository` over JPA.
- **Domain-event port + adapter** — model declares `DemandEvents`;
  `DemandEventsPropagation` (in `app-monolith`) fans events to projections + other
  contexts.
- **Domain "process" pattern** — `ShortagePredictionProcess` is a stateful
  policy/aggregate reacting to `onDemandChanged/onPlanChanged/onStockChanged`,
  recomputing via `ShortageForecasts`.

## Architecture & module layout

- Hexagonal / ports & adapters, applied **selectively** — only high-value complex
  logic sits inside the hexagon; simple CRUD does not.
- Physical enforcement by Gradle modules: pure `*-model` (no framework leakage,
  domain classes package-private) vs `*-adapters` (JPA, DAOs, REST, Spring
  `@Configuration`). `app-monolith` is the composition root + cross-context glue.
- Spring Boot 2.0.2, Data JPA, Data REST (+ HAL browser), Web, Cloud (Contract),
  Lombok; tests in Groovy/Spock, consumer-contract via Spring Cloud Contract.

## CQRS / events / outbox

- Explicit "Command Query CRUD Responsibility Segregation": three lanes — CRUD via
  Spring Data REST, complex Commands via the hexagon, complex Queries via read
  models.
- Command side: Spring Data REST `@RepositoryEventHandler` (`CommandsHandler`
  with `@HandleBeforeCreate/@HandleAfterCreate`) delegates to `DemandService`.
- Read side: persistent projections from domain events (`CurrentDemandProjection`,
  `DeliveryForecastProjection`) + a query-time-composed `StockForecastQuery`.
- Event delivery is **synchronous in-process** via `DemandEventsPropagation` — no
  broker. No Event Sourcing, no transactional Outbox.

## Architecture enforcement

- No ArchUnit / jMolecules / Spring Modulith. Boundaries enforced structurally:
  (1) separate Gradle modules so `*-model` can't depend on adapters, and
  (2) Java package-private visibility on aggregates, entities, and repository ports.

## What to borrow (Java Spring Boot DDD template)

- Split each context into a framework-free `*-model` module + `*-adapters` module,
  with a thin `app-monolith` composition root.
- Use package-private visibility **plus** module boundaries (not just annotations)
  to make illegal dependencies impossible; add ArchUnit/Spring Modulith on top
  (this repo predates them).
- Model the "three lanes" pragmatically: Spring Data REST for genuine CRUD, a
  hexagon only for high-value commands, explicit read-model projections for
  queries — don't force every feature through the aggregate.
- Keep application services thin (load → one domain method → save); push rules
  into aggregates + injected policy objects; immutable Lombok `@Value` VOs.
- Define a domain-event **port** in the model (`DemandEvents`), implement
  propagation in the app layer, so contexts integrate without the domain knowing
  Spring or transports.
- Adopt Event Storming → Design-Level → executable-model workflow and check the
  boards into the repo as living documentation.
