## Overview

A production-grade reference modular monolith with DDD + CQRS + Clean
Architecture. Five bounded contexts in one deployment unit but strictly
decoupled: each owns its data (separate schema), exposes one narrow public
interface, and communicates only asynchronously via integration events over an
in-memory bus. The lessons — module isolation, layered internals, explicit
invariant enforcement, reliable messaging via Outbox/Inbox — map cleanly onto
Java/Spring.

## Modules & internal layering

- Five modules: User Access, Registrations, Meetings, Administration, Payments
  (Payments uses event sourcing).
- Each module = Clean Architecture with three internal submodules (→ Java:
  packages or Gradle/Maven submodules): **Domain** (aggregates, entities, VOs,
  business rules; no infra deps), **Application** (CQRS handlers, domain events,
  integration events, internal commands), **Infrastructure** (data access, DB
  init, IoC composition root, event bus, background workers).
- A separate **IntegrationEvents** assembly per module publishes only the event
  contracts others may depend on — never implementation.
- Each module has a static `Initialize` run at startup that wires its composition
  root.

## Module boundary rules & enforcement

- Each module owns its data in a **separate DB schema**; cross-module joins/FKs
  forbidden — a module could move to its own database with no app changes.
- Modules communicate only asynchronously via integration events; direct
  cross-module method calls not allowed.
- Each module exposes one **facade interface** (e.g. `IMeetingsModule`) with just
  `ExecuteCommandAsync<T>(command)` / `ExecuteCommandAsync(command)` /
  `ExecuteQueryAsync<T>(query)`. All inbound access goes through this gateway.
- A module depends only on other modules' integration-event contracts.
- Boundaries enforced automatically by architecture tests (NetArchTest → **ArchUnit**
  in Java): Domain must not depend on Infrastructure; no module references
  another's implementation namespace.

## Application layer (CQRS)

- Strict CQRS: writes via commands (`ICommandHandler<TCommand>`) loading
  aggregates and invoking domain behavior; reads via queries
  (`IQueryHandler<TQuery,TResult>`) that **bypass the domain model** and run raw
  SQL against DB views (Dapper).
- Mediator (MediatR) decouples invoker from handler. **Java equivalent**: a
  command/query bus — a hand-rolled dispatcher over `CommandHandler<C,R>` beans
  keyed by type, or a library (Axon command bus). Spring's `ApplicationEventPublisher`
  covers in-process events.
- Cross-cutting concerns are **decorators around each command handler**, applied
  in order: Logging (correlation IDs) → Validation (FluentValidation) → UnitOfWork
  (commit + dispatch domain events). **Java equivalent**: handler decorators / AOP
  aspects / a bus interceptor chain; FluentValidation → Bean Validation (JSR-380).
- Internal commands represent deferred work triggered by inbox processing.

## Domain layer & business rules

- Aggregates are consistency boundaries; state is private, mutated only through
  intention-revealing methods.
- Low primitive obsession: primitives grouped into VOs (`MeetingGroupId`,
  `MoneyValue`, `MeetingTerm`). **Java**: immutable records / final classes with
  constructor validation.
- **Explicit invariant checking**: methods call `CheckRule(new SomeRule(...))`
  where rules implement `IBusinessRule`; a violated rule throws
  `BusinessRuleValidationException`. All business logic stays in the domain.
  **Java**: a `checkRule(BusinessRule)` helper on a base `AggregateRoot` that
  throws; each rule a small class with `isBroken()`/`message()`.
- Aggregates record domain events via `AddDomainEvent()`; dispatched during
  unit-of-work commit (same transaction).
- Payments uses event sourcing (SQL Stream Store): `Load(history)` replays,
  `Apply()`/`When(event)` rebuilds, `Version` increments.

## Integration events, Outbox/Inbox, consistency

- Cross-module messaging is asynchronous via integration events; modules accept
  eventual consistency.
- **Outbox** (at-least-once): within the command's single DB transaction, the
  aggregate AND integration events are written to an Outbox table atomically; a
  per-module background worker (Quartz) polls, publishes, marks processed.
  **Java**: outbox table + Spring `@Scheduled`/Quartz poller (or Debezium CDC).
- **Inbox** (at-least-once, idempotent): a received event is persisted to an Inbox
  table; a worker reads unprocessed rows, runs an internal command per event, marks
  processed — duplicates converge.
- Consistency verified in system tests via a "sampling" technique with timeouts
  (poll until the projection catches up).

## What to borrow (mapped to Java/Spring)

- **Module-per-context with a single facade**: one public `XxxModule` interface
  exposing only `executeCommand`/`executeQuery`; everything else package-private.
- **One schema per module, zero cross-module joins/FKs** — even on a shared
  PostgreSQL, so modules can later be extracted without rework.
- **Command/query bus with a decorator (or AOP) chain**: Logging → Validation →
  UnitOfWork; put the transaction boundary + domain-event dispatch in the outermost
  UnitOfWork step so events fire inside the same `@Transactional` commit.
- **Explicit `BusinessRule` + `checkRule`**: base `AggregateRoot.checkRule(rule)`
  throwing `BusinessRuleValidationException`, keeping invariants in the domain —
  more testable than scattered `if/throw` or Bean Validation on entities.
- **Value Objects over primitives**: Java records for typed IDs and money/date
  values, validated at construction.
- **Transactional Outbox + Inbox** for inter-module events; `ApplicationEventPublisher`
  for in-process domain events, outbox for cross-module ones.
- **Architecture tests as guardrails**: ArchUnit for layering, module isolation,
  naming, VO immutability — run in CI so boundaries can't erode.
