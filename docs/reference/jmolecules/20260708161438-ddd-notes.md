## Overview

Tiny, dependency-free Java libraries that let you express architectural and DDD
concepts directly in code as annotations and marker interfaces — intent becomes
explicit and machine-readable while domain code stays free of technical
framework dependencies. Tooling then consumes those markers to generate
framework boilerplate (ByteBuddy at build time), tune serialization
(Spring/Jackson at runtime), and enforce structural rules (ArchUnit,
jQAssistant). Coordinates under `org.jmolecules`, with `jmolecules-bom` for
version management (DDD line ~1.9.x).

## Concept libraries & annotations

- **`jmolecules-ddd` annotations**: `@AggregateRoot`, `@Entity`, `@ValueObject`,
  `@Repository`, `@Identity` (marks the identifier field), `@DomainEvent`,
  `@Service`.
- **`jmolecules-ddd` type interfaces**: `AggregateRoot<T, ID>`, `Entity<T, ID>`,
  `Identifier` (marker for ID types), `Identifiable<ID>`, and
  `Association<T, ID>` for making cross-aggregate references explicit (reference
  by ID, not object graph).
- **`jmolecules-events`**: annotations `@DomainEvent`, `@DomainEventPublisher`,
  `@DomainEventHandler`, `@Externalized`; interfaces `DomainEvent`, `Externalized`.
- **`kmolecules-ddd`**: Kotlin-idiomatic variant of the building blocks.

## Architecture abstractions

- **Layered** (`jmolecules-layered-architecture`): `@DomainLayer`,
  `@ApplicationLayer`, `@InfrastructureLayer`, `@InterfaceLayer`.
- **Onion** (`jmolecules-onion-architecture`): `@DomainModelRing`,
  `@DomainServiceRing`, `@ApplicationServiceRing`, `@InfrastructureRing`;
  simplified `@DomainRing`, `@ApplicationRing`, `@InfrastructureRing`.
- **Hexagonal** (`jmolecules-hexagonal-architecture`): `@Application`,
  `@PrimaryPort`, `@SecondaryPort`, `@PrimaryAdapter`, `@SecondaryAdapter`.
- **CQRS** (`jmolecules-cqrs-architecture`): `@Command`, `@CommandDispatcher`,
  `@CommandHandler`, `@QueryModel`.

## Integrations (ByteBuddy, Spring, JPA, ArchUnit)

- **`jmolecules-bytebuddy`**: build-time bytecode plugin that translates
  jMolecules markers into target-tech code — maps `@Entity`/`@AggregateRoot`/
  `@Identity` to JPA `@Entity`/`@Id`/`@Version` and generates boilerplate, maps
  `@Service` to Spring `@Service`. Keeps hand-written domain code annotation-light.
- **`jmolecules-spring`**: runtime Spring `Converter`s turning primitives
  (`String`, `UUID`) into `Identifier` types and back, an `AssociationResolver<T>`
  for Spring Data repositories to dereference `Association`s, plus Boot
  auto-config.
- **`jmolecules-jackson`**: Jackson module (auto-configured under Boot) that
  serializes single-attribute `@ValueObject`/`Identifier` types as their inner
  value, not nested objects.
- **`jmolecules-archunit`**: rule sets — `JMoleculesDddRules` (verifies
  aggregate/entity/identifier structure, e.g. aggregates only reference other
  aggregates via `Association` or their identifier, not direct object reference)
  and `JMoleculesRules`. Run as ordinary ArchUnit tests to fail the build.
- **jQAssistant plugin**: validates architecture/DDD rules against the scanned
  graph and renders PlantUML diagrams.

## Pairing with Spring Modulith

- Spring Modulith recognizes jMolecules stereotypes: annotated types are
  surfaced in Modulith's module model, documentation (Asciidoctor/PlantUML), and
  verification.
- Modulith enforces module boundaries; jMolecules makes the building blocks
  inside each module explicit — together you get inter-module isolation
  (Modulith) + intra-module tactical clarity (jMolecules). Common modular-monolith
  pairing.

## What to borrow (Java Spring Boot DDD template)

- Depend on `jmolecules-bom` + `jmolecules-ddd`; annotate tactical building blocks
  (`@AggregateRoot`, `@Entity`, `@ValueObject`, `@Repository`, `@Identity`,
  `@DomainEvent`) so DDD intent is explicit and self-documenting, with zero
  framework imports in the domain layer.
- Model cross-aggregate references with `Association<T, ID>` and typed
  `Identifier`s instead of direct object references — bakes "reference other
  aggregates by identity" into the type system.
- Wire `jmolecules-archunit` (`JMoleculesDddRules`) into the test suite as a CI
  gate so DDD structural invariants are enforced, not just documented.
- Add `jmolecules-bytebuddy` so JPA/Spring boilerplate is generated from domain
  annotations — persistence concerns stay out of hand-written domain code.
- Include `jmolecules-spring` + `jmolecules-jackson` so identifier/value-object
  types serialize cleanly at the API boundary and resolve through Spring Data.
- Pick and annotate one architecture style (hexagonal/onion/layered/CQRS) and
  pair with Spring Modulith for automated boundary verification + generated docs.
