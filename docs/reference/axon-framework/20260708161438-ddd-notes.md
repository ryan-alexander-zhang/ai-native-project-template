## Overview

A Java framework for evolutionary, event-driven systems built on DDD + CQRS +
Event Sourcing. Supplies ready-made building blocks — aggregate handling,
aggregate repositories, command/event/query buses, event stores, sagas — with
sensible defaults. Messaging sits at the core, giving location transparency so
the same annotated model runs in one JVM or distributed via Axon Server.
Integrates with Spring Boot (autoconfiguration starter) but doesn't require it.

## Aggregate programming model

- `@Aggregate` marks the aggregate root (Spring stereotype); without Spring you
  register it with a `Repository`/`AggregateFactory`.
- `@AggregateIdentifier` marks the id field (routes commands to the instance).
- `@CommandHandler` on a constructor **creates** the aggregate; on a method
  **handles** subsequent commands. Command handlers validate/decide — they do
  not mutate state directly.
- State changes are emitted via `AggregateLifecycle.apply(event)` inside command
  handlers; the event is routed back into the same aggregate.
- `@EventSourcingHandler` methods apply state mutations from events (on publish
  and during replay). **The only place state should change.**
- `@AggregateMember` models child entities within the aggregate boundary.
- `@CreationPolicy(CREATE_IF_MISSING)` lets a normal handler create without a
  dedicated constructor; `AggregateLifecycle.markDeleted()` retires an instance.

## Messaging: commands / events / queries

- Three message types: commands (intent, one handler), events (facts, many
  subscribers), queries (data requests).
- Buses: `CommandBus`, `EventBus` (an `EventStore` is an `EventBus` when event
  sourced), `QueryBus`.
- Gateways are the ergonomic client API over the buses: `CommandGateway`,
  `EventGateway`, `QueryGateway` (dispatch, futures, correlation, retries).
- Subscription queries stream initial result + live updates via
  `QueryUpdateEmitter`.
- Location transparency: handlers are addressed by message type, so dispatch is
  identical in-process or distributed.

## CQRS & projections

- Hard split between the command model (aggregates, write side) and query models
  (read side / projections).
- Read models built by `@EventHandler` projectors updating query-optimized
  stores; `@QueryHandler` serves reads from them.
- `@ProcessingGroup` groups event handlers under an Event Processor.
- Two processor styles: **Tracking** (async, own thread(s), persisted
  `TrackingToken`, supports replay + parallel segments) and **Subscribing**
  (runs in the publishing transaction).

## Event sourcing

- Aggregate state is rebuilt by replaying its event stream through
  `@EventSourcingHandler`. The `EventStore` is the append-only source of truth.
- `EmbeddedEventStore` over an `EventStorageEngine` (`JpaEventStorageEngine` /
  `JdbcEventStorageEngine`) for standalone; `AxonServerEventStore` for Axon Server.
- Snapshots via `SnapshotTriggerDefinition` (e.g. `EventCountSnapshotTriggerDefinition`)
  + `Snapshotter`; restore = latest snapshot + subsequent events.
- Tracking processors can reset their token to rebuild projections from full
  history (add read models, fix bugs) without touching the write side.

## Sagas / process managers

- `@Saga` coordinates long-running, multi-aggregate processes (process-manager
  pattern).
- `@SagaEventHandler(associationProperty = "...")` routes each event to the right
  saga instance; `@StartSaga` / `@EndSaga` (or `SagaLifecycle.end()`) bound the
  lifecycle.
- Commonly uses `DeadlineManager`/`EventScheduler` for timeouts and compensating
  actions; `SagaLifecycle.associateWith(...)` for association.

## What to borrow & when not to

- Borrow the **command/event/query separation** and the "decide vs. apply"
  discipline (handlers validate + emit events; a separate handler mutates state)
  — clean DDD even without the framework.
- Borrow the explicit **projection layer** (`@EventHandler` projectors + separate
  `@QueryHandler` read stores) — the read/write split is valuable in a modular
  monolith too.
- Borrow annotation-routed messaging (address handlers by message type) to keep
  modules decoupled and independently movable later.
- Borrow **sagas/process managers** for long-running cross-aggregate workflows
  with explicit start/end and compensation.
- **Caveat — complexity**: full Axon imposes event sourcing, eventual consistency
  write↔read, token stores, processor tuning. For a template that may only need a
  modular monolith over a relational store, that's a steep curve.
- **Caveat — ops & lock-in**: the real scalability story means running Axon Server
  (commercial Enterprise for HA); the standalone JPA/JDBC store works but you own
  its performance. **If you don't genuinely need ES/distribution, adopt the
  DDD/CQRS structure and light messaging conventions, not the whole framework.**
