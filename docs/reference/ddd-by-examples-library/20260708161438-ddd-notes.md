## Overview

A public-library lending system built to show DDD end-to-end, from Event
Storming through tactical code. Models patrons placing/canceling holds and
checking out books under real constraints (hold limits, restricted books,
overdue rules, 60-day checkout). Java + Spring, Vavr for functional/monadic
error handling, Spring Data JDBC for persistence. Deliberately shows two local
architectures matched to complexity: rich hexagonal for **Lending**, CRUD-like
for **Catalogue**.

## Bounded contexts & domain

- Two bounded contexts as top-level packages under `io.pillopl.library`:
  `catalogue` and `lending`.
- `catalogue` is intentionally simple/CRUD-like (`Book`, `BookInstance`,
  `ISBN`, `Catalogue`, `CatalogueDatabase`).
- `lending` is the complex core, decomposed into sub-modules: `book`, `patron`,
  `librarybranch`, `dailysheet`, `patronprofile`.
- Core rules: only researcher patrons may hold restricted books; regular patrons
  capped at 5 holds; patrons with too many overdue checkouts can't place holds;
  open-ended holds are researcher-only.

## Tactical patterns in code

- **Aggregate** — `lending/patron/model/Patron.java`: pure domain logic,
  package-private constructor, owns its policies + `OverdueCheckouts` +
  `PatronHolds`; methods (`placeOnHold`, `cancelHold`, `checkOut`) return Vavr
  `Either<Failure, Event(s)>` instead of mutating/throwing.
- **Aggregate as state machine** — the `book` aggregate is distinct types
  `AvailableBook`, `BookOnHold`, `CheckedOut` (all implement
  `lending/book/model/Book.java`); transitions produce new typed instances.
- **Value Objects** — `HoldDuration`, `CheckoutDuration`, `NumberOfDays`,
  `PatronId`, `BookId`, `LibraryBranchId`, `PatronInformation`, `ISBN`;
  immutable, Lombok `@Value`.
- **Domain Policy / Specification** — `lending/patron/model/PlacingOnHoldPolicy.java`
  is a `Function3<AvailableBook, Patron, HoldDuration, Either<Rejection, Allowance>>`;
  each business rule is a lambda constant and `allCurrentPolicies()` returns the
  composed list the aggregate iterates. Clean Policy/Specification example.
- **Domain Events** — `commons/events/DomainEvent.java` / `DomainEvents.java`;
  events like `BookPlacedOnHold`, `BookHoldCanceled`, `BookCheckedOut`,
  `MaximumNumberOhHoldsReached`, `BookInstanceAddedToCatalogue`.
- **Application Service / Command Handler** — `lending/patron/application/hold/PlacingOnHold.java`
  (+ `CancelingHold`, `ExpiringHolds`, `CheckingOutBookOnHold`,
  `RegisteringOverdueCheckout`): load aggregate via repository, invoke domain
  method, publish events; commands are explicit objects (`PlaceOnHoldCommand`,
  `CheckOutBookCommand`, `CancelHoldCommand`).
- **Repository ports in the domain package** — `book/model/BookRepository`,
  `patron/model/Patrons`, `patronprofile/model/PatronProfiles`,
  `dailysheet/model/DailySheet`; adapters in `infrastructure`
  (`BookDatabaseRepository`, `PatronsDatabaseRepository`, `SheetsReadModel`,
  `PatronProfileReadModel`).
- **Cross-context integration via event handlers** —
  `book/application/CreateAvailableBookOnInstanceAddedEventHandler` (Catalogue →
  Lending), `PatronEventsHandler`.

## Architecture & module layout

- Hexagonal (ports & adapters) within Lending; each sub-module follows `model`
  (domain + port interfaces) / `application` (use cases) / `infrastructure`
  (adapters, DB entities, Spring config) / `web` (controllers).
- Package-by-bounded-context first, then package-by-layer inside; dependencies
  point inward (`model` is framework-free).
- Shared kernel in `commons` (`aggregates/Version`, `AggregateRootIsStale`,
  `commands/Result`, `events/...`).
- Spring wiring isolated in `*Configuration`/`*Config` classes in infrastructure.

## CQRS / events / outbox

- **CQRS-lite**: explicit read models separate from write aggregates —
  `dailysheet` (holds-to-expire / checkouts-to-overdue, via `SheetsReadModel`)
  and `patronprofile` (`PatronProfile`, `CheckoutsView`, `HoldsView`, via
  `PatronProfileReadModel`, exposed by `PatronProfileController`). Read models
  query the DB directly, bypassing aggregates.
- Domain events drive eventual consistency between aggregates/contexts.
- **Outbox-style publishing**: `commons/events/publisher/StoreAndForwardDomainEventPublisher`
  + `EventsStorage` (persist-then-publish); also `JustForwardDomainEventPublisher`
  (immediate) and `MeteredDomainEventPublisher` (metrics decorator) — swappable.
- Optimistic concurrency via `commons/aggregates/Version`. No full event
  sourcing — state is stored, events are published.

## Architecture enforcement

- ArchUnit tests enforce boundaries, e.g. `model_should_not_depend_on_infrastructure`,
  `model_should_not_depend_on_spring`.
- No jMolecules and no Spring Modulith.

## What to borrow (Java Spring Boot DDD template)

- Package-by-bounded-context, then layer inside (`model`/`application`/
  `infrastructure`/`web`); repository **port interfaces in the domain package**,
  adapters in `infrastructure`, so dependencies point inward.
- Encode business rules as first-class composable **Policy/Specification** objects
  (a list the aggregate iterates) instead of scattered `if` checks.
- Return explicit result types (`Either`/`Try`/`Result`) from aggregate and
  application-service methods rather than throwing for domain rejections; keep
  aggregates immutable and model lifecycle as distinct types.
- Keep the domain framework-free and enforce it with **ArchUnit** in CI; a modern
  template can layer jMolecules/Spring Modulith on top for the same guarantees.
- Provide dedicated **read models** for queries (CQRS-lite) that hit the DB
  directly, separate from write aggregates.
- Ship a pluggable domain-event publisher with an **outbox** (store-and-forward)
  plus immediate and metered variants, so the consistency model is config, not a
  rewrite.
- **Match local architecture to complexity**: keep simple CRUD sub-modules thin
  (Catalogue), reserve full hexagonal for the complex core (Lending).
